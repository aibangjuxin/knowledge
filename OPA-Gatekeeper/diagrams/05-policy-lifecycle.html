OPA Gatekeeper Policy Lifecycle

Policy Authoring → Testing → Generation → Admission Enforcement

Phase 1 — Policy Authoring

  • • 每条策略独占一个目录
  • • src.rego — Rego 策略实现
  • • src_test.rego — 单元测试
  • • @kinds 装饰器指定 K8s 资源类型
  • • @parameter 装饰器定义可配置参数
  • • Policy Library 提供共享 Rego 规则

Phase 2 — Local Testing

  • • OPA CLI (gke-tep-resources GCS)
  • • make rego_test 运行全部测试
  • • 验证预期 violations == 实际 violations
  • • Gatekeeper 评估 violation rule
  • • 只要有 ≥1 violation 即 deny
  • • 完全本地验证，无需集群

Phase 3 — Konstraint Generation

  • • make constraints 触发 Konstraint
  • • 自动生成 ConstraintTemplate YAML
  • • 含 @kinds → 自动生成 Constraint CR
  • • 含 @parameter → 需手动创建 Constraint
  • • 55 条策略 → 49 个 L1 + 6 个 L2
  • • Konstraint 来源: gke-tep-resources GCS

Phase 4 — Admission Enforcement

  • • kubectl apply / GitOps 部署到 GKE
  • • API Server → Gatekeeper Webhook
  • • Gatekeeper 评估 matching Constraints
  • • L1: 强制 deny，不可覆盖
  • • L2: 团队在平台边界内可调
  • • L3: dryrun 或 Exemption 处理
