K8s Gateway NetworkPolicy — Tenant Traffic Flows

teamname-int 命名空间两种访问模式：直连容器 / Kong API 代理
Flow 1 — Direct Container Routing
客户端请求直接通过 Gateway HTTPRoute 路由到 teamname-int 命名空间内的普通容器 Pod（apigateway: NONE），无需 Kong 代理。
Client → ILB → abjx-gw Pod → HTTPRoute → Direct Container Pod
Flow 2 — Kong DP API Routing
客户端请求经过 Gateway HTTPRoute 到达 Kong DP（abjx-int-kdp），Kong DP 再代理到 teamname-int 命名空间内的 Kong API Pod（apigateway: KONG）。流量经 Kong 鉴权后仍回到源租户命名空间。
Client → ILB → abjx-gw Pod → HTTPRoute → Kong DP → Kong API Pod
Flow 3 — Microservice Intra-ns
teamname-int 命名空间内部，微服务 Pod（type: ms）可与同命名空间内的 KONG、NONE 容器互通，实现租户内微服务逻辑层的受控互通。
NONE ←→ KONG ←→ ms (同一命名空间内)
关键约束
Kong DP 代理的 API 请求最终必须回源到 teamname-int 命名空间内的 Kong API Pod，由 default-allow-kdp-ingress-to-kong-rt NetworkPolicy 控制入站规则（端口 8080/8443）。
Namespace: kong-apw-kong-int → teamname-int (Kong API Pod)

teamname-int NetworkPolicy 关键规则

Ingress（入站）
  • • default-allow-gw-ingress-to-no-gw-rt — 允许 abjx-gw-int → NONE Pod
  • • default-allow-kdp-ingress-to-kong-rt — 允许 kong-apw-kong-int → KONG Pod (8080/8443)
  • • allow-intra-ns-kong — KONG Pod 接收同类请求
  • • allow-intra-ns-nogateway — NONE Pod 接收 KONG/NONE 请求
  • • allow-intra-ns-ms — ms Pod 接收 KONG/NONE/ms 请求
Egress（出站）
  • • default-allow-dns — DNS 53 UDP/TCP → kube-system
  • • default-allow-egress-to-drn — 10.0.0.0/8 + DRN
  • • default-allow-egress-workload-identity — 127.0.0.1/32, 169.254.169.252/32:988
  • • default-allow-restricted-api — 199.36.153.4/30:443
  • • allow-intra-ns-kong — KONG Pod → KONG/NONE
  • • allow-intra-ns-nogateway — NONE Pod → NONE（禁止访问 KONG，防横向越权）
