# GKE DNS Resolution Flow — Full Debug Guide

> **Goal:** Trace a DNS query from a GKE Pod to the final answer — every hop, every decision point, and how to debug each layer when something breaks.

---

## 1. The Four DNS Layers in a GKE Pod

When your Pod needs to resolve a name, it doesn't just "ask the internet." It goes through a **multi-layer resolution chain**:

```
Layer 1: Pod /etc/resolv.conf
        ↓
Layer 2: Node Local DNS Cache (169.254.169.254)
        ↓
Layer 3: VPC DNS Resolver (169.254.254.254)  ← Cloud DNS managed resolver
        ↓
Layer 4: Cloud DNS Decision Engine
         ├── Response Policy (override)
         ├── DNS Peering Zone (pull from peer VPC)
         ├── Forwarding Zone (forward to target IPs)
         └── Default Recursion (ask public internet)
```

Understanding which layer is responsible for which queries is the key to debugging.

---

## 2. Pod /etc/resolv.conf — Where It Starts

Every Pod has a `/etc/resolv.conf` generated by kubelet based on the cluster's DNS policy:

```bash
# Inside any GKE Pod
cat /etc/resolv.conf
```

Typical output:

```
nameserver 10.x.x.x          ← ClusterIP of kube-dns/CoreDNS Service (NOT 169.254.254.254)
search <ns>.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```

**Wait — 10.x.x.x, not 169.254.254.254?**

This depends on your GKE cluster's **DNS policy**:

| DNS Policy | Nameserver in Pod | Notes |
|------------|-------------------|-------|
| `Default` (cluster-first) | `10.x.x.x` (kube-dns ClusterIP) | Query goes to CoreDNS in cluster |
| `ClusterFirstWithHostNet` | `10.x.x.x` (kube-dns ClusterIP) | Same as default, even with host network |
| `Default` + Cloud DNS as VPC DNS | `169.254.254.254` | **GKE manages this automatically** when VPC DNS is 169.254.254.254 |

**In practice (GKE with VPC-native clusters):**

The VPC resolver at `169.254.254.254` is the GCP-managed recursive DNS server for the VPC. GKE nodes are configured to use this as their DNS resolver. When a Pod queries `10.x.x.x` (kube-dns), the node's DNS proxy intercepts and routes to `169.254.254.254`.

**`ndots` explained:**

```bash
options ndots:5
```

If the query has **fewer than 5 dots**, Kubernetes appends search domains first:

```
Query: "my-svc"          → tries: my-svc.default.svc.cluster.local → kube-dns → VPC resolver
Query: "my-svc.default"   → same, search appended
Query: "my-svc.default.svc"  → has 4 dots, still search appended
Query: "my-svc.default.svc.cluster.local" → has 5 dots, query as-is first
```

This is why internal Service discovery works without fully-qualified names — but it also means queries with few dots go through the full search path before timing out.

## 3. Node Local DNS Cache — Optional but Recommended

GKE can run a **node-level DNS cache** on each node. This is a local instance of `coredns` or `unbound` running on the node itself:

```
Pod sends query → Node Local DNS Cache (127.0.0.1:53 on the node)
                     │
                     ├── cache hit → return immediately
                     └── cache miss → forward to VPC DNS resolver (169.254.254.254)
```

**Benefits:**
- Reduces DNS query latency (local cache, no network hop)
- Reduces load on kube-dns/CoreDNS
- Reduces forwarding traffic to VPC resolver
- Improves performance for repeated queries

**Enable on GKE:**
```bash
gcloud container clusters update YOUR_CLUSTER \
  --zone=ZONE \
  --enable-node-local-dns-cache
```

**Verify it's running:**
```bash
# On a GKE node (via debug node)
kubectl run -it --rm debug --image=busybox --restart=Never -- sh
# Inside the pod:
nslookup kubernetes.default
# Check which IP it resolves from
```

## 4. VPC DNS Resolver — 169.254.254.254

This is GCP's **managed DNS resolver for the VPC**. It is not a real IP on the network — it is an anycast IP handled by GCP's DNS infrastructure.

```
169.254.254.254 = VPC-level recursive DNS resolver
Accessible from: any VM, GKE node, or Pod in the VPC
Does NOT require: any special configuration in your Pod
```

**What it does:**

1. Receives DNS queries from workloads in the VPC
2. Checks against Cloud DNS **managed zones** associated with this VPC
3. Routes based on match (Response Policy → Peering → Forwarding → Recursion)
4. Returns the answer

**Key property:** This resolver is **shared by the entire VPC** — all subnets, all nodes, all Pods. There is one resolver per VPC.

## 5. Cloud DNS Decision Engine — The Four Mechanisms

When the VPC resolver receives a query, it follows this **priority order**:

```
Priority 1: Response Policy
            ↓ (no match)
Priority 2: DNS Peering Zone
            ↓ (no match)
Priority 3: Forwarding Zone
            ↓ (no match)
Priority 4: Default Recursion → public internet
```

### 5.1 Response Policy (Highest Priority)

Intercepts queries and returns **custom answers** defined by you. No upstream query is made.

```
Query: "dev.internal.aibang"
  → Response Policy matches "dev.internal.aibang."
  → Cloud DNS returns: 10.0.50.100  (override)
  → Target DNS server NOT consulted
```

**Use cases:**
- Override specific records for testing
- Redirect internal domains to different IPs
- Blocklist certain domains

### 5.2 DNS Peering Zone

**Pulls** a DNS zone from a **peered VPC** over VPC Network Peering.

```
VPC A (your GKE VPC)
  └── Query: "svc.internal.corp"
       → DNS Peering Zone: "internal.corp."
            → Peered VPC (corp VPC) has authoritative zone for internal.corp.
            → Answer pulled from peer VPC's DNS
```

**Requirements:**
- VPC Network Peering must be established between both VPCs
- Both VPCs must have the Peering Zone configured (bidirectional)
- No VPN or interconnect needed — traffic stays within GCP's network

### 5.3 Forwarding Zone

**Pushes** queries to **target IP addresses** you specify.

```
VPC (your GKE VPC)
  └── Query: "api.internal.aibang"
       → Forwarding Zone: "aibang."
            → Forward to 10.0.100.53 (your internal Unbound/BIND server)
                 → Returns: 10.0.50.20
```

**Requirements for target DNS server:**
- Reachable from VPC (same VPC, VPC Peering, VPN, Interconnect)
- Listening on UDP/TCP port 53
- Firewall: allow UDP/TCP 53 from `169.254.0.0/16`
- Stable IP recommended (use ILB for HA)

### 5.4 Default Recursion

No matching zone? Cloud DNS resolves the domain **recursively via public internet**.

```
Query: "www.google.com"
  → No matching managed zone
  → Cloud DNS recurses: Root → TLD (.com) → google.com authoritative NS
  → Returns: 142.250.x.x
```

## 6. VPC Binding — Zones Are Inert Until Bound

This is a critical, often overlooked point:

> **Creating a Cloud DNS managed zone does NOT automatically make it resolve.** You MUST associate (bind) it with a VPC.

**Without VPC binding:**
```
Cloud DNS zone "aibang." created
  → Zone exists in Cloud DNS
  → But NO VPC uses it
  → Pod queries "svc.internal.aibang" → Cloud DNS → NXDOMAIN
```

**With VPC binding:**
```
Cloud DNS zone "aibang." created
  → Associated with VPC "my-vpc"
  → All queries from my-vpc matching "aibang." are intercepted
  → Zone is ACTIVE
```

**Check zone visibility:**
```bash
# Show which VPCs a zone is bound to
gcloud dns managed-zones describe ZONE_NAME \
  --format="yaml(visibility,forwardingConfig,peeringConfig)"
```

**Bind zone to VPC at creation:**
```bash
gcloud dns managed-zones create "aibang-internal" \
  --description="Internal aibang domain" \
  --dns-name="aibang." \
  --visibility=private \
  --networks=https://www.googleapis.com/compute/v1/projects/PROJECT/global/networks/MY_VPC
```

**Add VPC to existing zone:**
```bash
gcloud dns managed-zones update "aibang-internal" \
  --add-private-gcp-network=https://www.googleapis.com/compute/v1/projects/PROJECT/global/networks/MY_VPC
```

**List all zones and their VPC bindings:**
```bash
gcloud dns managed-zones list \
  --format="table(name,dnsName,visibility,description)"
```

## 7. Full Resolution Flow — Step by Step

### Scenario: GKE Pod resolves `api.internal.aibang`

```
┌─────────────────────────────────────────────────────────────────┐
│ STEP 1: Pod-initiated query                                     │
│ Pod: "I need api.internal.aibang"                               │
│ /etc/resolv.conf: nameserver 10.x.x.x, search ... ndots:5     │
│ Query has 4 dots → search domains appended first               │
└────────────────────────────┬────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────┐
│ STEP 2: Node Local DNS Cache (if enabled)                       │
│ Node intercepts query → checks local cache                     │
│ Cache hit? → return immediately                                 │
│ Cache miss? → forward to 169.254.254.254                       │
└────────────────────────────┬────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────┐
│ STEP 3: VPC DNS Resolver (169.254.254.254)                     │
│ Receives: api.internal.aibang                                  │
│ Checks: Is this zone in this VPC?                              │
│ Decision order:                                                 │
│   1. Response Policy?     → check rules...                     │
│   2. DNS Peering Zone?    → check peer VPC...                  │
│   3. Forwarding Zone?     → check targets...                    │
│   4. Default Recursion?   → ask public DNS...                   │
└────────────────────────────┬────────────────────────────────────┘
                             ↓
        ┌──────────────────────────────────────┐
        │  Zone "aibang." matches Forwarding   │
        │  Zone → Forward to 10.0.100.53       │
        └──────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────┐
│ STEP 4: Target DNS Server (10.0.100.53)                        │
│ Internal Unbound/BIND receives: api.internal.aibang           │
│ Looks up zone: internal.aibang                                 │
│ Returns: A record → 10.0.50.20                                 │
└────────────────────────────┬────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────┐
│ STEP 5: Response path                                           │
│ VPC resolver receives answer from 10.0.100.53                  │
│ Returns A → Node Local DNS Cache (caches result)               │
│ Returns A → Pod                                                 │
│ Pod gets: 10.0.50.20                                           │
└─────────────────────────────────────────────────────────────────┘
```

## 8. Layer-by-Layer Debug Guide

When DNS fails, work backwards from the outermost layer.

### Debug L0: Basic Reachability

```bash
# Can you reach the VPC DNS resolver from a Pod?
kubectl exec -it test-pod -- sh
nc -vz 169.254.254.254 53
# Should succeed: Connection to 169.254.254.254 53 port [tcp/*] succeeded!

# Or with dig:
kubectl exec -it test-pod -- dig +short +time=3 @169.254.254.254 google.com
# Should return IP if VPC DNS is working
```

### Debug L1: Pod /etc/resolv.conf

```bash
kubectl exec -it test-pod -- cat /etc/resolv.conf

# Verify:
# - nameserver is reachable
# - search domains are correct
# - ndots value matches your expectation

# Check search path — are namespaces included?
kubectl exec -it test-pod -- sh -c 'for domain in "" "default.svc.cluster.local" "svc.cluster.local" "cluster.local"; do echo "Query: api\$domain"; dig +short api\$domain @10.x.x.x; done'
```

### Debug L2: Node Local DNS Cache

```bash
# Check if node local DNS cache is enabled in your cluster
gcloud container clusters describe CLUSTER_NAME --zone=ZONE \
  --format="yaml(networkConfig)"

# Check resolv.conf on the node (not from pod, use node debugging)
# By default, if enabled, it intercepts queries to 10.x.x.x
```

### Debug L3: VPC DNS Resolver

```bash
# Test from a VM or Pod in the same VPC
# Use dig with explicit +trace to see full recursion
dig @169.254.254.254 api.internal.aibang +trace +answer

# Check NS records for the zone
dig @169.254.254.254 aibang. NS

# Check if the zone is actually visible from this VPC
gcloud dns managed-zones describe aibang-internal \
  --format="yaml(visibility,forwardingConfig,peeringConfig)"
```

### Debug L4: Layer-by-Layer Domain Tracing

For a domain like `api.dev.internal.aibang`, trace **each level**:

```bash
# Start from the ROOT (TLD nameservers)
dig @169.254.254.254 aibang. NS
# Expected: returns NS records for aibang. (from Cloud DNS or your target)

# Check if the zone exists in Cloud DNS
gcloud dns managed-zones list \
  --filter="dnsName:aibang."

# Check forwarding config
gcloud dns managed-zones describe aibang \
  --format="yaml(forwardingConfig)"

# Check if VPC is associated
gcloud dns managed-zones describe aibang \
  --format="yaml(visibility)"   # should be "private"

# Check VPC DNS config
gcloud compute networks describe MY_VPC \
  --format="yaml(dnsConfiguration)"
```

### Debug L5: Forwarding Target Reachability

```bash
# Can Cloud DNS reach your forwarding target?
# Test from a VM in the same VPC:
dig @10.0.100.53 api.internal.aibang

# Check if target DNS is listening on TCP/UDP 53
nc -vz 10.0.100.53 53

# Check firewall rules allow 169.254.0.0/16 → target:53
gcloud compute firewall-rules list \
  --filter="network:MY_VPC AND direction:INGRESS AND destinationRanges:10.0.0.0/8"

# Check return path — can target DNS reply to Pod CIDR?
# (important: UDP response might come from different source)
```

### Debug L6: Simulate Full Chain from Pod

```bash
# Run a debug pod
kubectl run -it --rm dns-debug --image=busybox --restart=Never -- sh

# Inside pod:
# Step 1: Query the VPC resolver directly
nslookup api.internal.aibang 169.254.254.254

# Step 2: Query kube-dns directly
nslookup api.internal.aibang kube-dns.kube-system.svc.cluster.local

# Step 3: Check what the ndots search would produce
nslookup api.internal.aibang.default.svc.cluster.local

# Step 4: Query forwarding target directly (from inside cluster)
# (only works if target is reachable from cluster)
nslookup api.internal.aibang 10.0.100.53
```

## 9. VPC Binding Checklist

Use this when a Cloud DNS zone isn't resolving:

```
□ Zone created in Cloud DNS
□ Zone visibility is "private" (not "public")
□ Zone is associated with the correct VPC
□ VPC has the zone's DNS name in its routing
□ Forwarding targets (if any) are reachable from VPC
□ Firewall allows UDP/TCP 53 from 169.254.0.0/16 to target IPs
□ No conflicting Peering Zone for the same domain prefix
□ Response Policy (if any) is correctly configured
```

**List all private zones and their VPCs:**
```bash
gcloud dns managed-zones list --filter="visibility=private" \
  --format="table(name,dnsName,visibility)"
```

**List VPC DNS configuration for your network:**
```bash
gcloud compute networks describe MY_VPC \
  --format="yaml(dnsConfiguration)"
```

## 10. DNS Priority Conflicts — What Wins?

Cloud DNS applies the **longest suffix match** among all active mechanisms:

```
Query: "api.dev.internal.aibang"

Matching zones (by longest suffix):
  Response Policy: "api.dev.internal.aibang."  → exact match → RETURNS OVERRIDE
  Response Policy: "dev.internal.aibang."       → parent match → SKIPPED
  Forwarding Zone: "aibang."                    → suffix match → NOT CHECKED (Response Policy wins)
```

**Priority order among different mechanisms:**

| Priority | Mechanism | Condition |
|----------|-----------|-----------|
| 1 | Response Policy | matches rule → return override |
| 2 | DNS Peering | matches zone in peered VPC |
| 3 | Forwarding Zone | forward to target IPs |
| 4 | Default Recursion | no match → public internet |

**Same mechanism, multiple zones:**
- Cloud DNS uses **longest suffix match** among Response Policies
- For Forwarding Zones: all matching zones evaluated, most specific wins

## 11. Real Debug Scenario: abc.def.hi.com

Let's say you need to resolve `abc.def.hi.com` and it's failing. Trace from right to left:

```bash
# Step 1: Verify the ROOT TLD (hi.com nameservers)
dig @169.254.254.254 hi.com. NS
# If this fails: VPC DNS can't reach public internet
# Check: VPC firewall, Cloud NAT, or if you have custom routing

# Step 2: Verify the second-level zone (def.hi.com)
dig @169.254.254.254 def.hi.com. NS
# If this fails: zone may not exist in any target DNS

# Step 3: Query the full domain against the VPC resolver
dig @169.254.254.254 abc.def.hi.com. A

# Step 4: Check if a Forwarding Zone exists for hi.com or def.hi.com
gcloud dns managed-zones list \
  --filter="dnsName:hi.com. OR dnsName:def.hi.com."

# Step 5: Check Response Policy rules
gcloud dns response-policies list \
  --filter="description:hi.com OR description:def.hi.com"

# Step 6: Check DNS Peering (if VPC is peering with another)
gcloud dns managed-zones list \
  --filter="visibility=private" \
  --format="yaml(name,dnsName,peeringConfig)"

# Step 7: If all above fail, query the forwarding target directly
dig @TARGET_DNS_IP abc.def.hi.com. A

# Step 8: Verify the zone exists in target DNS server
# (SSH to target DNS if you have access)
# Check zone file: is def.hi.com or hi.com configured?
```

## 12. Useful gcloud Commands Summary

```bash
# List all managed zones
gcloud dns managed-zones list

# Describe a specific zone (shows VPC bindings)
gcloud dns managed-zones describe ZONE_NAME

# Check forwarding targets
gcloud dns managed-zones describe ZONE_NAME \
  --format="yaml(forwardingConfig)"

# Check DNS Peering config
gcloud dns managed-zones describe ZONE_NAME \
  --format="yaml(peeringConfig)"

# Check Response Policies
gcloud dns response-policies list
gcloud dns response-policies describe POLICY_NAME

# Check VPC DNS configuration
gcloud compute networks describe VPC_NAME \
  --format="yaml(dnsConfiguration)"

# View Cloud DNS query logs (requires Cloud Logging)
gcloud logging read \
  'resource.type="dns_query" AND query_name="abc.def.hi.com."' \
  --limit=20 --order=desc

# Test from a specific VPC (using VM)
gcloud compute ssh VM_NAME --zone=ZONE -- \
  "dig @169.254.254.254 abc.def.hi.com +short"
```
