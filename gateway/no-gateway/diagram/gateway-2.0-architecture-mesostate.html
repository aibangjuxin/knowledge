Gateway 2.0 Architecture — Multi-Tenant GKE Gateway (Mesostate)

Left: Entry Flow (PSC → IIP) | Right: Multi-Tenant GKE Gateway — one GKE Gateway per team, all in namespace abjx-int-gkegw

Entry Flow: PSC → IIP

  • • Client → Cross-Project PSC Attachment (Producer: B Project)
  • • PSC → IIP (Internal Ingress Proxy) — this project as entry
  • • IIP performs Header / Path validation
  • • IIP routes by Host header to team-specific GKE Gateway
  • • teamN.appdev.aibang → /teamN/* path

Multi-Tenant GKE Gateway (abjx-int-gkegw)

  • • All teams share namespace: abjx-int-gkegw
  • • One GKE Gateway CR per team (team1/2/3-gateway)
  • • Each team has: HTTPRoute + Listener 443 + TLS Policy
  • • ILB assigned per gateway (10.100.0.101/102/103)
  • • Team domains: *.appdev.aibang

Per-Team Upstream Choice

  • • Team 1: Kong Gateway 2.0 (namespace: kong-team1)
  • • Team 2: Direct Runtime (abjx-int-gkegw)
  • • Team 3: Kong Gateway 2.0 (namespace: kong-team3)
  • • Switch upstream by updating HTTPRoute upstreamRef
  • • Kong: Auth/Rate/Logging plugins per team namespace

IIP Routing Rules

  • • Host-based routing: teamN.appdev.aibang
  • • Path rewrite: /teamN/* → upstream route
  • • Header injection for upstream identification
  • • Rate limiting + WAF at IIP layer
  • • Mutual TLS (mTLS) to GKE Gateway

Control Plane & Security

  • • Kong: KongAdmin API / Deck sync per team namespace
  • • GKE Gateway: kubectl / Gateway API CRs
  • • TLS: *.appdev.aibang wildcard cert
  • • mTLS between IIP and upstream gateways
  • • NetworkPolicy per namespace (tenant isolation)

Adding a New Team (N+1)

  • 1. Create teamN-gateway GKE Gateway CR in abjx-int-gkegw
  • 2. Create teamN-http-route HTTPRoute + Listener + TLS
  • 3. IIP config: add Host rule teamN.appdev.aibang
  • 4. Choose upstream: Kong (kong-teamN) or Direct Runtime
  • 5. For Kong: create namespace kong-teamN + deploy
