ASM Solo Ambient Mesh — 2-Segment TLS with ztunnel HBONE

一段 TLS 终结(L1/L2 不变) + 一段 ztunnel 节点级 mTLS(L3 应用无感) · Solo Distribution 1.30.2-solo-fips · 对照 background.md

段 1 · Client → Gateway(不变)

  • ListenerSet 上 tls.mode: Terminate 不变
  • wildcard TLS Secret 不变(在 abjx-listenerset-int)
  • Gateway GW Class = `istio` 不变
  • Gateway Pod 内部还是 Envoy sidecar(zatmbient 不管 L7)
  • 跨平台方向的边界划界完全跟 background.md 一致

段 2 · Gateway → Pod(根本改变)

  • ztunnel DaemonSet 在每节点抓包,运行 HBONE 15008
  • 跨节点流量 ztunnel↔ztunnel 双跳 SPIFFE mTLS
  • 节点↔pod 仍然走 ambient CNI 创建的 iptables 初始 redirect
  • Pod 内部仅 listen HTTP :8080,无证书 mount,无 443
  • HTTPRoute / Service 形态不变,只是流量加密层透明了

架构红利(为什么值得切换)

  • DR 删除:不再配置 trafficPolicy.tls.mode = ISTIO_MUTUAL
  • 应用回归 HTTP:不再 mount 证书、不再 skip-verify
  • 资源瘦身:每个 Pod 省一份 Envoy sidecar(典型 -50~100m CPU / -128Mi mem)
  • 跨租户接入样式 3 YAML(Service/Deploy/HTTPRoute),从 4 降到 3
  • SPIFFE 自动:istiod 在 mesh 内部自动签发,跨 ns 零配置

踩过的坑 / 反模式(必读)

  • ⚠️ Platform NS 也打 ambient → Gateway 内部 Envoy 被 Zatmbient 干扰
  • ⚠️ DR 残留 trafficPolicy.tls → 路由冲突,502
  • ⚠️ Pod 上 istio-proxy 还在 → 重启 deployment 让 webhook 重评
  • ⚠️ NetPol 默认 deny 阻断 HBONE 15008 / Envoy 15021 / 15090
  • ⚠️ GKE Dataplane V2 与 Istio CNI 冲突 → 改用 V1 或 chianed=true
