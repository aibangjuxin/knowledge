Release Workflow — caep-infra-gcp-release

Branch model (main / dev / feature) · Commit subject CAEP-xxxx: · PR Approve group · 4 gates · CR with main commit id · 11-step end-to-end
feature / dev branch / code step
flow / success / prod release
CR / audit / non-prod execute
dev→main promotion / metadata
gate / fix / blocking signal

Phase 1-2 — Develop & Merge to dev

  • step 1-2: feature/CAEP-xxxx-* + self-test gate 1CI
  • step 3-4: PR feature→dev via PR Approve groupGATE 2
  • step 3: squash merge · linear history on dev
  • step 4: post-merge review (no conflict / no secrets)

Phase 3 — Non-prod Execute & Fix loop

  • step 5: jumpbox / UAT · run.sh dev mode
  • step 6: verify fail → fix feature (commit prefix preserved)
  • step 7: redo 2-3-4-5 until dev HEAD verify = green
  • output: real artifacts/<r>-setup-*.log + *-verify-*.log

Phase 4 — Promote / CR / Prod release

  • step 8: PR dev→main (merge commit, no squash)GATE 3
  • step 9: raise CR — MUST reference main HEAD SHA
  • step 10: CR Reviewer · release notes = git logGATE 4
  • step 11: run.sh --mode=prod EXPECTED_SHA= self-checks before deploy

FOUR GATES — single source of truth

GATE 1 · local CI (step 2)
GATE 2 · PR Approve group (step 3)
GATE 3 · PR dev→main (step 8)
GATE 4 · CR Reviewer approves (step 10)
STEP 11 · prod SHA self-check (defence-in-depth, runtime)
All four gates are real checkpoints (branch protection hooks or human approver actions); step 11's runtime SHA self-check is defence-in-depth and cannot replace any gate.
