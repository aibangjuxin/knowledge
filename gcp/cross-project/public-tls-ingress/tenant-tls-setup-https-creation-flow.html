Resource Creation Flow — Producer + Consumer 顺序

从 cert 出发 → 5 个 Producer 资源 → PSC 桥接 → 4 个 Consumer 关联操作 → e2e 验证

同 swim-lane: Producer 在上, Consumer 在下; 跨 VPC 唯一桥梁 = Service Attachment (Producer) → PSC NEG (Consumer)

SSL / Proxy / FR (cyan)

顺序依赖 (emerald solid)
PSC 跨 VPC (amber dashed)
gcloud bug 走 import (rose)

Producer 5 步 (顺序强依赖)

  • • ① cert → 没有任何依赖 (可与 ② 并行)
  • • ② HC → 不依赖 (但 update BS 引用它, 见 §3.3)
  • • ③ Proxy → 依赖 ① cert + 已有 UM
  • • ④ FR → 依赖 ③ Proxy + 已有 IP (10.0.1.4)
  • • ⑤ SA → 依赖 ④ FR + PSC NAT subnet

Consumer 4 步 (含 2 个 update)

  • • ① BS (import) → 重建为 HTTPS, 不依赖 Producer
  • • ② NEG → 依赖 Producer ⑤ SA (--psc-target-service)
  • • ③ add-backend → 把 NEG 挂到 BS
  • • ④ attach armor → 把现成 Cloud Armor 挂回新 BS
  • • 2 & 3 步骤之间需要等 ~60s 让 SA auto-accept

踩坑提醒 (gcloud CLI)

  • • ④ FR 和 ① BS 必须用 import + YAML，create 报错因 multi-word 值 (INTERNAL MANAGED / EXTERNAL MANAGED) 解析 bug (见 §6.2)
  • • ② NEG 必须在 BS 改完 protocol 之后重建 (老 NEG connection state 失效, 见 §6.4)
  • • ① cert 文件名有下划线: tenant.taobao.caep.uk_bundle.crt (见 §6.1)
