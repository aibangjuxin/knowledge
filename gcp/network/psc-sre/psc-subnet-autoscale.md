### 一、 为什么网络层 100% 不会中断？（验证文档的正确性）

文档里提到扩容“不会中断流量”，这是正确的。我为你拆解一下底层的运作原理：

1.  **GCP 原生支持包含关系的“原地扩容”**
    在 GCP 中，只要新的子网掩码比原来的小（/25 < /27），且**新的网段完全包含旧的网段**，GCP 是允许直接执行 `expand-ip-range` 操作的。
    `192.168.240.0/27` 完全被包含在 `192.168.240.0/25` 内，所以满足数学条件。

2.  **已分配的 NAT IP 会发生什么？**
    什么都不会发生。之前分配给 PSC Service Attachment 的 IP（比如 `.2`、`.3`）依然老老实实地保留在原地。GCP 底层的虚拟交换机只是在路由表中把这个子网的边界从 `.31` 推到了 `.127`。

3.  **现有的 TCP 连接会断吗？**
    **绝对不会。** GCP 的底层使用的是分布式的状态连接跟踪（Connection Tracking / Conntrack）。已建立的 TCP 会话（源IP、源端口、目标IP、目标端口 4元组）会被保存在状态表中。扩容只是给池子加了新水，老水里的鱼毫无感知。新进来的请求一旦发现前 28 个 IP 的端口快用完了，就会自动去拿 `.32 ~ .127` 的新 IP 来做 SNAT。

### 二、 纠正与补充：文档遗漏的 2 个“生产级巨坑”

虽然文档理论上没错，且推荐了“新增子网追加到 Attachment”作为最佳实践。但既然你明确问了**“原地扩容”**（把 `/27` 改为 `/25`），我必须指出真实操作中的坑：

#### 坑 1：Terraform / IaC 的“销毁重建”风险（极易引发重大事故）
如果你是用 Terraform 管理这段子网的：
```hcl
resource "google_compute_subnetwork" "psc_nat" {
  name          = "psc-nat-core01-att01"
  ip_cidr_range = "192.168.240.0/27" # 你想把这里改成 /25
  # ...
}
```
**🚨 致命警告：** 在某些旧版本的 Terraform GCP Provider 中，修改 `ip_cidr_range` 可能会被标记为 **`Forces replacement` (强制替换)**！
一旦 Terraform 识别为替换，它的动作是：
1. 强制删掉旧的 /27（此时正在跑的 NAT IP 瞬间灰飞烟灭，**业务立刻 100% 宕机**）。
2. 创建新的 /25。
3. 重新绑定给 Attachment。

**正确的规避动作：**
*   执行前**必须**跑 `terraform plan`，仔细看是不是 `~ update in-place`（原地更新）。
*   如果是原地更新，大胆敲 yes，不会掉一包数据。
*   如果是 `-/+ replace`，绝对不能执行！必须用 gcloud 命令行手动扩容，然后再把 Terraform state 拉平。

*(注：较新的 Google Provider 已经支持了安全的 in-place 扩容，但生产环境防备这一手极其重要)。*

#### 坑 2：后端防火墙与对端服务的白名单
你把子网从 `/27` 扩大到 `/25` 以后：
*   假设之前你在 GKE 后端应用的防火墙里，或者中间件（Kong/Nginx）的 IP 限制策略里，只放行了 `192.168.240.0/27`。
*   扩容后，网络本身没断，但新进来的连接拿到了 `192.168.240.40` 这个新 IP。
*   **结果：** 这个新 IP 去了后端，被防火墙拦住了。业务表现为：部分请求成功，部分请求报错（Connection Refused / Timeout）。
