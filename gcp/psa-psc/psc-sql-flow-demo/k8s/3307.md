## §1 探索起点:Pod 内 `ss` 看不到 3307 的困惑

### §1.1 第一反应

直觉上:"如果 proxy 要连 3307,那应该在 Pod 里能看到 3307 在 listen 吧?"

```bash
$ kubectl exec POD -c cloud-sql-proxy -- ss -nltp
# 没有 3307 — 完全空
```

这让人怀疑:**到底有没有东西在用 3307?**

### §1.2 第二反应:看进程的实际命令

```bash
$ kubectl exec POD -c cloud-sql-proxy -- ps aux
USER  PID  COMMAND
65532 1    /cloud-sql-proxy --psc --structured-logs --auto-iam-authn \
                  --address=127.0.0.1 --port=5432 <instance>
```

读 cmdline 看到 `--port=5432`,所以 proxy 在 pod 内 listen 5432,不是 3307。**Pod 内真的没有 3307 在 listen — 这反而是正常的。**

### §1.3 第三反应:看错地方了

那 3307 出现在哪?去看 proxy 日志:

```
2026-07-06T11:02:56.494Z [dev:asia-east2:sql-dev-01] failed to connect to instance:
Dial error: failed to dial: dial tcp 192.168.64.104:3307: i/o timeout
```

**这个 3307 不是 Pod 内 listen 端口,是 proxy 出 pod 去 dial 的目标端口(192.168.64.104:3307)**。

**关键转折点**:

> **`ss` 看不到 3307 完全正常。3307 在 Pod 内从来就不是 listen 端口,是出 pod 的目标端口。**

## §2 探索转折:3307 是怎么定出来的?

### §2.1 想知道 3307 是谁写死的

CLI 上 `--port=5432` 是我配的(`address=127.0.0.1` 也是)。但日志里出现的 `3307` 我完全没配过。哪来的?

检索 [Cloud SQL Private Service Connect 文档](https://cloud.google.com/sql/docs/postgres/about-private-service-connect),在 PostgreSQL 那段看到:

> "The supported serving ports for PostgreSQL are as follows:
> * TCP port **5432** for direct connections to PostgreSQL database server.
> * TCP port **6432** for direct connections to PgBouncer server when using Managed Connection Pooling.
> * TCP port **3307** for connections through Cloud SQL Auth Proxy."

**这 3 个数字都是 Cloud SQL 服务端固定的 serving port,不是 client 配置的。**

**这 3 个数字都是 Cloud SQL 服务端固定的 serving port,不是 client 配置的。**

### §2.2 完整 serving port 表(权威版 — 不要简化)

GCP 官方文档 [Private Service Connect 服务端口表](https://cloud.google.com/sql/docs/postgres/about-private-service-connect#psc-backend)对每个引擎都列出 PSC Service Attachment 上**同时 listen** 的几个端口:

| 引擎 | 引擎直接服务端口 | PgBouncer(Managed Connection Pooling) | Auth Proxy 专用 | 服务端总共 listen 几个 port |
| --- | --- | --- | --- | --- |
| **PostgreSQL** | **5432**(裸 PG wire protocol) | **6432**(PgBouncer) | **3307**(Cloud SQL Auth Proxy) | **3 个** |
| **MySQL** | **3306**(裸 MySQL wire protocol,含 Managed Connection Pooling) | — | **3307**(Cloud SQL Auth Proxy) | **2 个** |
| **SQL Server** | **1433**(裸 TDS) | — | **3307**(Cloud SQL Auth Proxy) | **2 个** |

**关键事实**:

1. **服务端 listen 多个 port,不是"只" listen 3307**(只是**3307 是为 Auth Proxy 流量预留的**)。
2. **每个端口对应一种连接模式,模式之间是互斥的**:
   - 同一条 TCP 连接上,要么走"裸协议"(5432/3306/1433),要么走"通过 Auth Proxy"(3307)。
   - 5432 / 3306 / 1433 这三个"裸端口"也有用 Managed Connection Pooling(PgBouncer)走 6432 的场景。
3. **3307 与 5432/3306/1433 不互通**:
   - 你 dial 5432,服务端就给 PostgreSQL wire protocol 回应(走 SCRAM/PG-native 鉴权)
   - 你 dial 3307,服务端就知道"这是从 Auth Proxy 进来",看 token 不看密码

### §2.2.5 修正之前说法(避免误导)

之前的简化表达 "**Cloud SQL 服务端只 listen 3307**" 严格来讲是错的 — 准确的说法是:

> ✅ **"对于 Auth Proxy 这条连接路径,Cloud SQL 服务端在 3307 这个端口提供服务"**
>
> ❌ ~~"Cloud SQL 服务端只 listen 3307"~~

| 错的简化 | 对的精确 |
| --- | --- |
| "服务端只听 3307" | "**Auth Proxy 这条连接路径**走 3307 这个端口" |
| "3307 是 Cloud SQL 唯一的端口" | "3307 是服务端**为 Auth Proxy 流量预留**的端口;其他端口(5432/3306/1433)另有用途" |

**Lex 这次追问非常关键** — 因为这个澄清直接决定 NetworkPolicy 要不要也开 5432/6432/3306:

- 如果你 **只用 Auth Proxy**(proxy 帮应用走 3307) → **只开 3307**
- 如果你 **混合架构**(有些应用直连 5432,有些走 Auth Proxy 3307) → **5432 + 3307 都要开**
- 如果你有 **PostgreSQL Managed Connection Pooling** → **加开 6432**

详见 §6.0 决策表。

### §2.3 验证:192.168.64.104 哪来的

```bash
$ gcloud sql instances describe sql-dev-01 --project=dev --format=json \
    | jq '.settings.ipConfiguration.pscConfig'
{
  "pscEnabled": true,
  "pscAutoConnections": [
    {
      "consumerNetwork": "projects/dev/global/networks/dev-vpc",
      "ipAddress": "192.168.64.104",     # ← 这个就是日志里的 IP
      "status": "READY"
    }
  ]
}
```

**`192.168.64.104` 是 PSC endpoint 的 IP,从 SQL Admin API 的 `pscConfig.pscAutoConnections.ipAddress` 来**。proxy 启动时调这个 API 拿到,然后才知道该 dial 哪个 IP。

**注意**:同一个 PSCEP IP(192.168.64.104)**所有这几个端口都 serve** — 你 dial `192.168.64.104:5432`、`192.168.64.104:6432`、`192.168.64.104:3307` 是**三条不同的 TCP 连接**,由服务端按 port number 分流到不同处理路径。

### §2.4 链路拼起来看

```
[部署侧: 你配的]
  --port=5432                 ← Pod 内 proxy listen 的端口 (ss 看得见)
  --psc                       ← 关键标志:走 PSC
  INSTANCE_CONNECTION_NAME    ← dev:asia-east2:sql-dev-01

                │
                ▼

[proxy 启动时调 SQL Admin API]
  GET sqladmin.googleapis.com/sql/v1/projects/dev/instances/sql-dev-01
                │
                ▼

[Admin API 返回 instance metadata]
  pscConfig.pscAutoConnections[0].ipAddress = "192.168.64.104"

                │
                ▼

[proxy 用 --psc 标志自动选 serving port]
  端口 3307   ← Cloud SQL 服务端固定,client 改不了

                │
                ▼

[proxy 每次新连接 dial 这个 endpoint]
  dial tcp 192.168.64.104:3307     ← 这就是日志里看到的那 3307
```

**3307 = "服务端写死的约定 + proxy 调 Admin API 查到的 IP"组合。客户端零配置。**

## §3 探索深化:身份验证在 client 端还是 server 端?

### §3.1 文档检索

Lex 又深入了一步:**"auth proxy 做身份验证 — 这个'验证'在哪做?客户端(client)还是服务端(server)?"**

检索 [Cloud SQL IAM Authentication 文档](https://cloud.google.com/sql/docs/postgres/iam-authentication),关键原话:

> "Automatic IAM database authentication lets you **hand off requesting and managing access tokens to an intermediary** Cloud SQL connector, such as the Cloud SQL Auth Proxy or one of the Cloud SQL Language Connectors. With automatic IAM database authentication, users need to pass only the IAM database username in a connection request from the client. **The connector submits the access token information for the password attribute on behalf of the client.**"

> "IAM database authentication uses OAuth 2.0 access tokens, which are short-lived and valid for only one hour. **Cloud SQL connectors are able to request and refresh these tokens**..."

### §3.2 client vs server 分工

| 验证环节 | 谁做 | 在哪 |
| --- | --- | --- |
| **OAuth 2.0 token 申请** | **客户端** (proxy 用 GSA + WI / metadata server) | Pod 内 |
| **token 注入到 PG password 字段** | **客户端** (proxy `--auto-iam-authn`) | Pod → PSCEP 之间 |
| **token 真伪验证** | **服务端** (Cloud SQL 调 IAM API) | Cloud SQL 实例进程里 |
| **DB-level GRANT 决定权限** | **服务端** (PG `GRANT` 表) | Cloud SQL 实例进程里 |

**"Auth Proxy 做身份验证" 字面上看像服务端做,实际是客户端做的工作多 + 服务端做的工作少:**

- **客户端 (proxy)** 做:dial target 拼装、token 申请、token 注入、TLS 握手
- **服务端 (Cloud SQL)** 做:token 验证、最终裁定
- **端口 3307 = 客户端去向服务端的那条通道,决定了认证资料能不能送达**

### §3.3 Lex 简化表达

> "**我应该关心我的服务端侦听的地址**(192.168.64.104:3307),**而不是关心我 Pod 内 listen 的端口**。"

**心智公式**:

```
app 在 Pod 内连 本地端口    → NS 不参与 (loopback)
proxy 出 Pod 连 服务端端口  → NS 必须放行那个端口 (egress)
                        ↑
                你配的 NetworkPolicy
                永远只看这一跳
```

## §4 探索回到原问题:为什么必须开 3307?

### §4.1 三层防线,dial timeout 卡在哪一层

Lex 看到的报错:

```
Dial error: failed to dial: dial tcp 192.168.64.104:3307: i/o timeout
```

`i/o timeout` 几乎只意味着一件事:**包出了 Pod 到 VPC 路由,但服务端没在应到时间内响应**。

可能拦包的地方有 3 层:

| 层 | 谁拦? | 怎么看? | 怎么修 |
| --- | --- | --- | --- |
| **A. NS NetworkPolicy** | 你的 NS 是 deny-all,没配 3307 egress | `kubectl get netpol -n NS` | 加 `port: 3307` egress |
| **B. VPC Firewall** | GCP 项目级防火墙 | `gcloud compute firewall-rules list` | 加 firewall rule |
| **C. PSC Service Attachment** | service attachment 没起/没 ready | `gcloud sql instances describe ... \| jq .pscConfig.status` | 等就绪 / 重启 |

**Lex 当前 99% 卡在 A 层**(deny-all + 没配端口)。

### §4.2 验证是不是 A 层

直接绕开 proxy 在 Pod 内试 TCP 三步握手:

```bash
$ kubectl exec POD -c app -- \
    bash -c 'timeout 5 bash -c "echo > /dev/tcp/192.168.64.104/3307" \
            && echo TCP_OK || echo TCP_BLOCKED'

# 期望:TCP_OK    → netpol 已经放行 → 不是 A 层
# 期望:TCP_BLOCKED → 还是 A 层
```

`/dev/tcp/<ip>/<port>` 是 bash 内建,distroless 镜像可能没这个 — 但因为 Pod 内 proxy sidecar 一定有 bash(否则 proxy 也跑不起来)或者你直接用 app 容器内的 shell 测就行。

### §4.3 修复 NetworkPolicy(A 层)

**Lex 的现状**:NS deny-all + app=db 的 egress 但**端口没配置**。

加下面这条 egress:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-db-egress-3307
  namespace: <你的 NS>
spec:
  podSelector:
    matchLabels:
      app: db          # 匹配你的 db-app
  policyTypes: ["Egress"]
  egress:
    # DNS
    - to: []
      ports:
        - {protocol: UDP, port: 53}
        - {protocol: TCP, port: 53}

    # Cloud SQL PSC endpoint:必须放 3307
    # IP 来自 gcloud sql instances describe 的 pscConfig.pscAutoConnections.ipAddress
    - to:
        - ipBlock:
            cidr: 192.168.64.104/32     # ← 收紧到具体 PSCEP
      ports:
        - {protocol: TCP, port: 3307}   # ← 服务端 listen

    # cloud-sql-proxy 调 SQL Admin API + IAM(必须放)
    - to: []
      ports:
        - {protocol: TCP, port: 443}
```

验证:

```bash
$ kubectl apply -f netpol-allow-3307.yaml

# 再跑一次 TCP 验证(应该变 TCP_OK)
$ kubectl exec POD -c app -- \
    bash -c 'timeout 5 bash -c "echo > /dev/tcp/192.168.64.104/3307" \
            && echo TCP_OK || echo TCP_BLOCKED'
# → TCP_OK

# 再看 proxy 日志(应该不再 timeout)
$ kubectl logs -c cloud-sql-proxy --tail=30
# → 不再有 "failed to dial ... i/o timeout"
```

## §5 最终答案

### §5.1 一句话

> **3307 必须放,** 是因为 **Cloud SQL PSC Service Attachment 服务端在 3307 这个端口上接收 Auth Proxy 流量**。proxy 这个 client 替你的应用去 dial `192.168.64.104:3307`,client 端这边没法选这个数字。**NetworkPolicy 永远只看 Pod 出 Pod 的目标服务端端口,跟本地 listen 谁无关。**

**注意**:服务端不止 listen 3307 这一个端口 — 它还 listen 5432/3306/1433/6432 这些"直连"用的端口(详见 §2.2)。**只是 3307 是为 Auth Proxy 流量预留的**,你这个场景走 Auth Proxy,所以"今天你关心的那个数字"恰好是 3307。

### §5.2 完整版

**3 句话拆解**:

1. **本地端口随便**(proxy 在 pod 内 listen 5432 也好、3307 也好,随便配,NetworkPolicy 不过这一段因为 loopback 不算 egress)
2. **目的端端口写死** — Cloud SQL PSC Service Attachment 上 **每个引擎 listen 的端口是固定的**:
   - PostgreSQL:5432 / 6432 / 3307
   - MySQL:3306 / 3307
   - SQL Server:1433 / 3307
   - **3307 在所有引擎上都是同一个语义:为 Cloud SQL Auth Proxy 流量预留的端口**(GCP 服务端焊死的,客户端没法选)
3. **NS egress 必须放目的端那个端口**(3307 不在白名单里 = `dial tcp ...:3307: i/o timeout`,跟你 pod 内 listen 谁无关)

### §5.3 完整的"本地 vs 远端"对照表

| 端口号 | 出现位置 | 角色 | 谁 listen/dial | 怎么定的 |
| --- | --- | --- | --- | --- |
| `5432` (Pod 内 loopback) | Pod 内 | app → proxy 连接的目标 | proxy 在 `127.0.0.1:5432` listen | 你的部署 `--port=5432` |
| `5432` (Cloud SQL 服务端) | 出 pod → Cloud SQL | 直连 PG 用的端口 | Cloud SQL 服务端 listen | 服务端固定 |
| `3307` (Cloud SQL 服务端) | 出 pod → Cloud SQL | Auth Proxy 承载端口 | Cloud SQL 服务端 listen | 服务端固定 |
| `3307` (Pod 内 loopback) | **不存在** | — | — | 你可以让 proxy listen 任意本地端口(5432/3307/65535 都行) |
| `443` | 出 pod → Google APIs | proxy 拿 metadata/token/cert | 多个 Google 服务 | Google 维护 |

### §5.4 你(部署侧)+ GCP (服务端)各自的职责边界

| 你要做的(部署侧) | GCP 替你做的(服务端) |
| --- | --- |
| 给 proxy 选本地端口 → `--port=5432`(随便)| 把 Cloud SQL 实例 3 个 serving port 焊死 → 5432/6432/3307 |
| 给 KSA → GSA 绑定 (Workload Identity) | 给 Cloud SQL 实例开 PSC + 配 pscConfig |
| **开放 NS NetworkPolicy egress 3307**(给目标 server)| 在 PSC Service Attachment 端 listen 3307 |
| 开放 egress 443(给 SQL Admin API / IAM)| proxy 用 OAuth token 来找我,我只验证 |
| 选 Cloud SQL PG / MySQL / SQLServer 引擎 | proxy 根据 instance engine 自动选 `portConfig` 兜底 |

**你不能决定的事**:

- 服务端 listen 哪个端口(GCP 服务端固定)
- dial 哪个 IP(从 Admin API 自动查)
- token 申请(must 由 proxy 在 client 端做)

**你必须决定的事**:

- proxy 在 pod 内 listen 哪个端口(loopback 不影响 NS)
- NS egress 放哪个目的端口(必须与服务端 listen 对齐 → **3307**)
- NS egress 放哪个目的 IP(收紧到 PSCEP `192.168.64.104/32` 最安全)

### §5.5 所以 — 你原问题的所有版本

| 你问 | 现在怎么答 |
| --- | --- |
| "我 Pod 内完全看不到 3307 的关联" | 对,3307 不在 Pod 内 listen,只在你 Pod 出 pod 那一跳出现 |

| "Cloud SQL Proxy 的身份验证是哪里完成的?" | 客户端 + 服务端分工:**proxy 在 Pod 里申请 + 注入 token;Cloud SQL 服务端只验证 token** |
| "我 Pod 里没有 3307,但要连 3307 才算 cloud sql auth proxy" | 对。3307 不是"唯一"端口,而是**"走 Auth Proxy 流量对应的服务端端口"**。Pod 内 ss 看不到 3307 是因为 3307 不是 listen 端口,是 proxy 出 pod 的 dial target |

### §5.5.5 你的核心追问:"是否只 listen 3307?"的最终澄清

> **不是。**
>
> Cloud SQL 服务端 listen 的端口集合 = **引擎固定 + Auth Proxy 预留**,不同引擎端口数不同:
>
> - **PG 服务端 listen 3 个**:5432(裸协议)+ 6432(PgBouncer)+ 3307(Auth Proxy)
> - **MySQL 服务端 listen 2 个**:3306(裸协议)+ 3307(Auth Proxy)
> - **SQL Server 服务端 listen 2 个**:1433(裸 TDS)+ 3307(Auth Proxy)
>
> **3307 是 3 个引擎都共用的"Auth Proxy 专用 serving port"**,这是它的特殊性。但它不是"唯一"端口 — 服务端同时也开 5432/6432/3306/1433 接受走 PG/MySQL/SQL Server 原生 wire protocol 的连接。
>
> 为什么只有 3307 这个数字被反复提及?**因为 Auth Proxy 是 GCP 强烈推荐(从 [GKE 连接 Cloud SQL 文档](https://cloud.google.com/sql/docs/postgres/connect-kubernetes-engine))的生产路径**,所以"Proxy 出 pod 用 3307"这个数字最容易跟 NetworkPolicy 冲突。直连 5432/3306/1433 在 GKE Pod 场景里反而少用。

### §5.6 三个关键 chart 数(必背)

| 数字 | 含义 | 谁定的 | 怎么定的 |
| --- | --- | --- | --- |
| **`5432`** | 你 pod 内 proxy listen 的端口(PG 默认) / Cloud SQL 服务端 listen 裸 PG 协议的端口 | **两边都定**,两端都可以选 | Pod 内:你的 deployment `--port=5432`;服务端:GCP 固定 |
| **`3307`** | Cloud SQL 服务端 listen Auth Proxy 流量的端口(服务端固定);proxy 出 pod 时 dial 这个端口 | **服务端定**,客户端不能选 | GCP 服务端为 Auth Proxy 流量预留 |
| **`443`** | proxy 调 SQL Admin API / IAM API 用的端口 | **Google 服务定** | googleapis.com 域名使用 |

**NetworkPolicy 永远按"目的 server listen 端口"配**:

- **必有 3307**(若你走 Auth Proxy 流量)
- **必有 443**(proxy 必调 Admin API / IAM)
- **可选加 5432 / 6432 / 3306 / 1433**(若同一 NS 还有别的走直连或 Managed Connection Pooling 的 Pod)

### §5.7 两个权威证据(Lex 自查后提供的最终定型依据)

Lex 在文档前面心里嘀咕"3307 在 Pod 内看不到任何关联,为啥 egress 必须开 3307?" — 这一节的 2 个权威证据,就是用来最终定格这个问题的:

#### 证据 #1 — Google 官方文档原话(已确认存在)

[Cloud SQL SQL Proxy 总览](https://cloud.google.com/sql/docs/mysql/sql-proxy)第 4 节关于 3307 的原话:

> "You must allow outgoing (or egress) TCP connections to ports **443** and **3307**."
>
> "**While Cloud SQL Connectors can listen on any port, they create outgoing (or egress) connections to your Cloud SQL instance only on port 3307.**"
>
> "If your client machine has an outbound firewall policy, then make sure it allows outgoing connections to port 3307 on your Cloud SQL instance's IP address."

[Cloud SQL connect-auth-proxy](https://cloud.google.com/sql/docs/mysql/connect-auth-proxy)同样确认:

> "If you have an outbound firewall policy, make sure it allows connections to port 3307 on the target Cloud SQL instance."
>
> "**Note: The Cloud SQL Auth Proxy uses 3307 to connect to the Cloud SQL Auth Proxy server.**"
>
> "Port 3306 is the default port for the MySQL protocol over a direct connection."

**这段话对应的就是 Lex 的总结**:

> "**3307 是 outbound destination port, not a local port。**"

—— Google 明确说"Cloud SQL Connectors 可以 listen 任何本地端口(127.0.0.1:5432/3306/3307 任意),但**只在 3307 这个目的端口**创建到 Cloud SQL 实例的出站连接"。

**重要纠正**:Google 原话是 **"create outgoing connections only on port 3307"** —— 注意 **"only on port 3307" 是修饰 "outgoing connections"**,意思是"出 pod 的连接只用 3307 这一个目的端口",**不是** "服务端只 listen 3307"。仔细区分:
- "outgoing connections (from client) only use port 3307" — ✅ 我之前解释的"客户端 outbound 目的端口固定 3307"
- "server only listens on 3307" — ❌ 这才是错的简化的说法(详情见 §2.2.5)

#### 证据 #2 — GitHub issue #2337(用户实景重现)

[cloud-sql-proxy/issues/2337](https://github.com/GoogleCloudPlatform/cloud-sql-proxy/issues/2337) 是 GitHub 上的一个公开 issue,标题"**Cloud SQL Proxy with PSC `dial tcp 10.250.250.5:3307: i/o timeout`**"。摘要:

- **用户配置**:`cloud-sql-proxy PROJECT_ID:REGION:INSTANCE_NAME --psc`(跟 Lex 一样带 `--psc`)
- **完整步骤**:
  1. Create a cloudsql psql instance with psc
  2. Create a VM instance with cloud-sql-proxy
  3. Launch cloud-sql-proxy
  4. Try to connect to the database through the proxy
- **报错**:
  ```
  [PROJECT_ID:REGION:INSTANCE_NAME] failed to connect to instance:
  Dial error: failed to dial (...): dial tcp 10.250.250.5:3307: i/o timeout
  ```
- **用户困惑**:"The ip resolution seems ok, but the port used is not right. How can I configure the proxy to connect on the `5432` port ?"
- **结论**: **即使加 `--psc`,proxy 仍然用 3307 去 dial instance** — 用户想要"用 5432 直连"的请求被 GCP 设计**主动拒绝**(3307 是为 Auth Proxy 流量固定的 serving port,跟底层 engine 是 PG/MySQL 都一样)。

**这个 issue 就是 Lex 这次踩坑的"镜像案例"**:

| Lex 的场景 | issue #2337 场景 |
| --- | --- |
| `cloud-sql-proxy ... --psc --port=5432` | `cloud-sql-proxy ... --psc` |
| 报错 `dial tcp 192.168.64.104:3307: i/o timeout` | 报错 `dial tcp 10.250.250.5:3307: i/o timeout` |
| 困惑 "Pod 内看不到 3307,为啥必须开 3307?" | 困惑 "ip 解析对,端口用错(应该 5432 吧)?" |
| **Lex 想要的解释** | Google 文档已经说过了:Auth Proxy 路径只走 3307 |

**issue 的真正含义**(就是 Lex 现在搞清楚的):

1. **proxy 没有"用一个端口直连"模式**:它只有"用 Auth Proxy 出 3307"或"用 Auth Proxy 出别的 serving port"两种(后者实质上不可配)。
2. **IP 看上去对、PCAP 抓到 5432 也在 listen — 但你必须 dial 3307**:这是 Cloud SQL 服务端 served ports 决定,不是 IP 决定。
3. **GCP 不让客户端配这个数字**:即使你写 `--port=5432`(本地 listen 端口),proxy 仍然出 pod dial 3307。

#### §5.7 收尾:为什么这两个证据足以"完全解决疑惑"

这两条证据构成**权威三角**:

```
                 ┌──────────────────────────┐
                 │  Google 官方文档原话 (#1) │
                 │  "only on port 3307"      │
                 │  (outgoing connections)   │
                 └────────────┬─────────────┘
                              │
              "3307 是 outbound destination port"
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
        ▼                     ▼                     ▼
  出 pod 的 TCP        服务端 listen 端口       Auth Proxy
  dial target          是 Cloud SQL SA           是 client-side
  ← 你要放 NS           ← 服务端写死             (proxy 进程)
  egress 3307           ← 不归你管              ← 在你 Pod 里
                              │
                 ┌────────────┴─────────────┐
                 │  GitHub issue #2337 (#2)  │
                 │  user ran --psc →        │
                 │  still dialed ...:3307   │
                 │  → 实景证实              │
                 └──────────────────────────┘
```

- **文档原话(#1)** 告诉理论:Cloud SQL Connectors 的 outbound 永远走 3307
- **issue (#2)** 告诉实证:即使有 `--psc`,proxy 仍然 dial `10.250.250.5:3307` — 完全跟 Lex 的报错 `dial tcp 192.168.64.104:3307` 同模式
- **`cloud-sql-proxy` 默认行为 + `--psc` 标志** 告诉机制:本地 listen 端口任意,但 outbound destination port 固定 3307

**Lex 既然这两条已经把问题定性了,文档后续章节应该用这两条作为"确定答案"的引文来源,而不再是探索过程的中间观点。**

#### 唯一不能从这两条直接得到的细节

- 服务端 listen 多个端口(5432/3306/1433 等)"直连"用,但 Lex 当前场景走 Auth Proxy 所以是 3307 — 这部分还是靠 §2.2 的 PSC 服务端口表来支撑。

### §6.0 NetworkPolicy 端口决策表(放哪几个 → 看这一段就够了)

先回答你的具体问题:**3307 不够,要看场景组合**。以下是完整决策:

| 你的部署场景 | 必须开的 egress 端口 | 备注 |
| --- | --- | --- |
| **PG + 只用 Auth Proxy**(Lex 当前现实)| **3307** + **443** | 完整方案 |
| **PG + 仅直连 PG(裸协议)绕过 proxy** | **5432** + **443**(如果直连也走 IAM,还要 `cloudsql.instances.get`) | TLS 强制 |
| **PG + 直连 + Auth Proxy 混合** | **3307 + 5432 + 443**(最安全覆盖)| 详见 [`sql-3307-template.md`](./sql-3307-template.md) |
| **PG + Managed Connection Pooling (PgBouncer)** | **6432 + 443** | 端点是 PgBouncer,不走 Auth Proxy |
| **MySQL + 只用 Auth Proxy** | **3307 + 443** | 跟 PG 同 |
| **MySQL + 直连** | **3306 + 443** | |
| **MySQL + Managed Connection Pooling (built-in,不走 PgBouncer)** | **3306 + 443**(MySQL 的 pooling 走 native port) | 跟 PG 不同 |
| **SQL Server + Auth Proxy** | **3307 + 443** | |
| **SQL Server + 直连** | **1433 + 443** | |

**最大覆盖(推荐 — 任何混部都安全)**:

```yaml
egress:
  - to:
      - ipBlock:
          cidr: <PSCEP>/32  # 例如 192.168.64.104/32
    ports:
      - {protocol: TCP, port: 3306}   # MySQL direct
      - {protocol: TCP, port: 3307}   # Auth Proxy (any engine)
      - {protocol: TCP, port: 5432}   # PostgreSQL direct
      - {protocol: TCP, port: 6432}   # PgBouncer pooling
      - {protocol: TCP, port: 1433}   # SQL Server direct

  - to: []
    ports:
      - {protocol: TCP, port: 443}    # SQL Admin API / IAM
      - {protocol: UDP, port: 53}     # DNS
      - {protocol: TCP, port: 53}     # DNS
```

**收紧方案(只服务当前场景)** — Lex 当前用这个就够:

```yaml
egress:
  - to:
      - ipBlock:
          cidr: 192.168.64.104/32
    ports:
      - {protocol: TCP, port: 3307}   # 仅放 Auth Proxy 一个
  - to: []
    ports:
      - {protocol: TCP, port: 443}
      - {protocol: UDP, port: 53}
      - {protocol: TCP, port: 53}
```

## §6.1 完整的 NetworkPolicy 修复 yaml(直接能 apply)

```yaml
# netpol-allow-cloud-sql-3307.yaml
# 适用场景:NS 是 deny-all,Pod 用 cloud-sql-proxy sidecar 走 PSC 模式
# 必修:egress TCP/3307 → 目的 IP = PSCEP
# 推荐:egress TCP/443 → 让 proxy 能调 SQL Admin API

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-cloudsql-auth-proxy-egress
  namespace: <你的 NS>          # 替换
  labels:
    app: db
spec:
  podSelector:
    matchLabels:
      app: db                    # 匹配你的 db-app
  policyTypes: ["Egress"]

  egress:
    # ─────── 1. DNS (基本) ───────
    - to: []
      ports:
        - {protocol: UDP, port: 53}
        - {protocol: TCP, port: 53}

    # ─────── 2. cloud-sql-proxy → Cloud SQL PSC:3307 (必) ───────
    # PSCEP IP 192.168.64.104 来自 gcloud sql instances describe
    # 的 .settings.ipConfiguration.pscConfig.pscAutoConnections[0].ipAddress
    - to:
        - ipBlock:
            cidr: 192.168.64.104/32     # ← 收紧到 PSCEP IP(推荐)
            # 也可放宽到 PSC subnet,但收紧更安全
      ports:
        - {protocol: TCP, port: 3307}   # ← Cloud SQL 服务端 fixed serving port (Auth Proxy 专用)

    # ─────── 3. cloud-sql-proxy → SQL Admin API + IAM (必) ───────
    - to: []
      ports:
        - {protocol: TCP, port: 443}

    # ─────── 4. (可选) 老部署 / 兼容性 ───────
    # 如果你的集群里有老 Pod 走 5432/6432 直连,也放,免得混部翻车
    # - to:
    #     - ipBlock:
    #         cidr: 192.168.64.0/24
    #   ports:
    #     - {protocol: TCP, port: 5432}
    #     - {protocol: TCP, port: 6432}
```

应用 + 验证:

```bash
$ kubectl apply -f netpol-allow-cloud-sql-3307.yaml

# TCP 三步握手验证(绕开 proxy 直接测)
$ kubectl exec POD -c app -- \
    bash -c 'timeout 5 bash -c "echo > /dev/tcp/192.168.64.104/3307" \
            && echo TCP_OK || echo TCP_BLOCKED'
TCP_OK

# proxy 日志不再 timeout
$ kubectl logs POD -c cloud-sql-proxy --tail=20
# 预期看到 oauth token 申请 / TLS handshake / 连接成功

# 业务流验证
$ kubectl exec POD -c app -- curl -s http://localhost:8080/healthz
# 预期:正常业务
```
