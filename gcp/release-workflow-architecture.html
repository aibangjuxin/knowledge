Release Workflow — CAEP Infra GCP

14 steps · 4 phase gates · 4-layer branch topology (main / release/* / dev / feature) · CR + jumpbox gates

Phase 1 — Plan & Develop

  • step 1-2: branch scaffolding on top of main
  • step 3: cut feature/CAEP-xxxx-* off dev
  • step 4-5: Conventional Commits + local self-test
  • step 6: PR to dev with 1 approver
  • step 7: preflight.sh on jumpbox (Gate 1)

Phase 2 — Non-prod Execute

  • step 8: squash merge feature → dev
  • step 9: setup-uk.sh + verify-uk.sh
  • step 10: fix-bugs loop (returns to step 4 by name)
  • step 11: raise CR linking release/vX.Y.Z SHA
  • Gate 2: verify-green + CR-submitted (parallel)

Phase 3 — Notes & PR Main

  • step 12: summary.md from real artifacts
  • step 12 cont.: PR release/vX.Y.Z → main
  • step 13: CR approval gate (external system)
  • final review by release owner on notes
  • CR-rejected → back to step 3 (loop red)

Phase 4 — Prod Release & Close

  • 14a: squash merge PR → main
  • 14b: git tag -a caep-infra-vX.Y.Z
  • 14c: prod release on jumpbox
  • 14d: close CR in CR system
  • 14e: +7d retrospective → next release's README §7

Rollback & Hotfix

  • verify fail → rollback-<region>.sh dry-run
  • new dir releases/vX.Y.Z-rollback/ (never overwrite)
  • CR-rejected: close PR, split or fix
  • prod alert > 24h → hotfix/vX.Y.Z+1
  • hotfix: Conventional Commits + same workflow from step 5

Baseline Contract Preserved

  • ✅ main = production change
  • ✅ dev = non-prod change
  • ✅ feature branch = development
  • ✅ commit log rules (jira + PR-only)
  • ✅ PR merge protection (approve group)
