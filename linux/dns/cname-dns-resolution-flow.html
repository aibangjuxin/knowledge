DNS 链 (3 步解析)

  • • ① Query — Client ask api1.team1.caep.uk via Resolver
  • • ② CNAME — Cloudflare wildcard: *.team1.caep.uk → tenant-lb.platform.caep.uk
  • • ③ A — tenant-lb.platform.caep.uk → 34.105.229.97 (Tenant Project GLB 入口)
  • • 关键点: CNAME 与 A 记录不能共存于同一 hostname, 所以 A 挂在别名上
  • • 解耦: LB IP 改变只改 ③, 不动 ②; 新增 team / API 只加 ② 的一条 CNAME

SNI / Host header 透传

  • • TLS SNI = api1.team1.caep.uk (原始域名, 非 CNAME 别名)
  • • HTTP Host = api1.team1.caep.uk (原始域名)
  • • 证书: GLB cert SAN 必须包含 *.team1.caep.uk 或更宽的 *.caep.uk
  • • Nginx / Kong: 监听原始域名, 不是 CNAME 别名 — 详见 apply-cname.md §二.3
  • • URL Map host rule: GLB 用 api1.team1.caep.uk 匹配, 转发到对应 backend

职责分离 (本图 vs public-tls-ingress)

  • • 本图 (cname-dns-resolution): ZONE 0–2 完整 + ZONE 3 浅色 入口示意
  • • public-tls-ingress: ZONE 3 深色 Producer 侧 (ILB → MIG 内部 TLS 链) 完整实现
  • • DNS 修改最小化: 切环境 / 迁移 LB / 灰度发布, 改 1 条 A 记录即可
  • • 跨 project 解耦: caep.uk 域名不绑定到任何具体 project 的内部资源
