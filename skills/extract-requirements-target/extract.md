# 提取需求与目标

## 概述

阅读源材料，区分信号与背景信息，生成仅包含两个核心输出的紧凑摘要：`Requirements`（需求）和 `Target`（目标）。
保持用户意图不变，压缩重复内容，避免虚构不存在的需要。

## 提取工作流

### 1. 识别文档类型

- 判断输入是技术方案、架构说明、需求文档、会议总结、产品简报还是混合草稿。
- 假设关键事实可能分散在不同章节、列表、表格或非正式笔记中。
- 忽略修饰性语言、状态更新和背景说明，除非它们改变了范围或意图。

### 2. 提取 `Requirements`（需求）

- 提取必须完成、交付、满足、支持或避免的内容。
- 包含明确的需求，如功能要求、技术约束、依赖关系、成功标准、截止日期、兼容性期望、合规规则或非功能性需求。
- 将重复表述合并为一行简洁的条目。
- 优先使用具体表述而非泛泛重述。
- 保持每条需求独立且可快速浏览。

### 3. 提取 `Target`（目标）

- 推断作者想要达成的最终状态。
- 将目标表述为最终的业务、技术、交付或决策目标。
- 区分目标与实现细节：
  - `Requirement` 回答"需要什么或受到什么约束？"
  - `Target` 回答"本文档试图达成什么结果？"
- 优先使用一个简短段落或 1 到 3 个紧凑要点。

### 4. 谨慎处理模糊信息

- 仅在源材料暗示但未直接说明时，将条目标记为推断。
- 不要将假设升级为事实。
- 如果文档包含多个竞争目标，区分：
  - 主要目标
  - 次要目标
- 如果源材料过于模糊，说明目标不明确，并列出最可能的解读。

## 压缩规则

- 将长段落缩减为短小、信息密集的陈述。
- 移除示例、轶事、理由和历史背景，除非它们实质性地影响需求或目标。
- 当更短的表达更清晰时，避免复制用户的原文。
- 在术语对领域至关重要时，保留源材料中的用词。
- 保留数量、版本、截止日期、环境和命名系统等信息。

## 输出格式

默认使用以下结构：

```markdown
## Requirements
- ...
- ...

## Target
...
```

如果源材料复杂，添加一个可选章节：

```markdown
## Notes
- Inferred: ...
- Missing detail: ...
```

## 提取启发式规则

- 将"need"、"must"、"should"、"expect"、"goal"、"want"、"deliver"、"support"、"avoid"和"success"视为强需求信号。
- 将"so that"、"in order to"、"最终"、"目标"、"希望"和"achieve"视为强目标信号。
- 当文档混合问题陈述和解决方案建议时，从问题中提取需求，从预期结果中提取目标。
- 当文档高度技术化时，保留平台名称、环境名称、接口和运营约束。

## 质量标准

- 使结果易于理解，无需重新阅读原始文档。
- 优先完整性而非过度简化，但避免将摘要重新变成完整文档。
- 保持最终输出足够简洁，让团队成员能在一分钟内理解需求。

## 示例触发语

- "从这份 RFC 中提取真正的需求和目标。"
- "阅读这份技术说明，只给我需求和目标。"
- "这份文档太长了。总结需求和目标。"
- "从这份方案里提炼需求和目标，不要展开解释。"
