# 文件到需求（Files To Requirements）

## 概述

当输入不是单一清晰文档，而是包含笔记、文档、配置、草稿、日志、Markdown 文件或混合工作材料的本地文件夹时，使用此技能。
目标不是 RAG 风格的问答，而是将零散文件转化为明确的需求声明，包含明确的差距和后续问题。

## 何时使用

- 用户指向本地目录而非单一源文件。
- 源材料分散在多个文件或嵌套文件夹中。
- 任务是澄清用户需求、约束条件以及仍然缺失的信息。
- 用户希望在总结或需求分析之前有一个可重复的预处理工作流。

**不要使用此技能的情况：**

- 用户已提供一个清晰文档，仅需简短的 `Requirements` 和 `Target` 提取。
- 任务是通用知识检索或基于知识库的开放式问答。

## 工作流程

### 1. 收集文件

- 从用户提供的目录或文件集开始。
- 按以下条件过滤：
  - 目录范围
  - 文件扩展名
  - 修改时间
  - 可选的文件名关键词
- 优先高信号文本源，如 `md`、`txt`、`rst`、`json`、`yaml`、`yml`、`csv`、源代码注释和轻量级文档。
- 排除明显低价值文件，如二进制文件、锁文件、构建产物、vendor 文件夹、缓存和生成的包，除非用户明确要求。
- 记录使用的收集规则，以便结果可审计。

### 2. 提取文本和元数据

- 对每个选定文件，保留：
  - 绝对路径或仓库相对路径
  - 标题或推断主题
  - 修改时间（如可用）
  - 简短摘要
  - 关键信号：目标、约束、决策、TODO、开放问题或接口
- 通过引用源文件为重要声明保留证据锚点。
- 如果文件过大，仅总结影响范围、约束或预期输出的部分。

### 3. 识别主题和重叠

- 按主题、系统、功能、项目阶段或利益相关者关注点对文件分组。
- 将重复项和近似重复项合并为一个标准化声明。
- 分离：
  - 重复事实
  - 冲突声明
  - 过时或被替代的材料
- 当两个文件表达相同内容时，优先选择最清晰和最新的来源。
- 如果两个来源冲突且时间先后无法解决，保留两者并明确标记冲突。

### 4. 生成需求候选项

- 将整合后的知识转换为以下维度的候选需求：
  - 用户目标
  - 功能需求
  - 技术约束
  - 交付或工作流预期
  - 缺失依赖或缺失决策
- 将每个候选项表达为简短、可审查的声明。
- 标记每个项目的状态：
  - explicit（明确的）
  - inferred（推断的）
  - unresolved（未解决的）
- 除非源材料明确强制，否则不要将实现想法作为需求呈现。

### 5. 生成澄清问题

- 将歧义转化为简短的具体后续问题列表。
- 问题应聚焦于会实质性影响范围、架构、输出或顺序的决策。
- 优先可以用一句话或一小撮选项回答的问题。
- 如果问题较多，按主题分组，但保持简洁。

### 6. 生成最终需求简报

- 生成结构化摘要，使团队成员无需重读源文件即可使用。
- 保持简报稳定且基于模板。
- 明确区分已知信息、推断信息和仍需确认的信息。

## 默认输出

默认使用以下结构：

```markdown
## 收集范围
- 路径：
- 过滤器：
- 已审阅文件：

## 源文件清单
- `path` | 主题 | 修改时间 | 简短摘要

## 整合知识
- 主题：
- 确认事实：
- 冲突或重复项：

## 需求候选项
- [明确的] ...
- [推断的] ...
- [未解决的] ...

## 澄清问题
- ...
- ...

## 最终需求简报
### 目标
...

### 需求
- ...

### 约束
- ...

### 缺失信息
- ...
```

## 工作规则

- 始终以文件为依据，而非猜测。
- 优先少量强需求声明，而非冗长复述。
- 保持重要或争议性内容的来源可见性。
- 如果源文件集噪声较大，说明哪些文件驱动了最终结论。
- 如果源文件集不完整，直接说明，不要用假设填补空白。

## 与其他技能的关系

- 当真正的问题是零散源收集和规范化时，在使用 [`extract-requirements-target`](/Users/lex/git/knowledge/skills/extract-requirements-target/SKILL.md) 之前使用此技能。
- 当输入已是单一文档或干净的粘贴文本块时，单独使用 `extract-requirements-target`。

## 典型触发场景

- "扫描这个本地文件夹，告诉我真正的需求是什么。"
- "收集这些 Markdown 和 YAML 文件，合并重叠笔记，然后澄清需求。"
- "我有一堆本地文档，把它们变成结构化知识，然后输出需求澄清问题。"
- "先整理本地目录里的文件，再帮我把需求澄清出来。"
