# nginx `ngx_http_sub_module` 评估

## 1. 目标与问题定义

你想做的事情，本质上不是“单纯开启一个 Nginx 编译模块”，而是通过：

```nginx
sub_filter_once off;
sub_filter "__CSP_NONCE__" $request_id;

proxy_cache_bypass 1;
proxy_no_cache 1;
```

来实现下面这个目标：

- 上游应用返回 HTML / JS 模板时，先写一个占位符，例如 `__CSP_NONCE__`
- Nginx 在响应返回给客户端前，把这个占位符替换成每个请求唯一的值，例如 `$request_id`
- 同时通过禁用缓存，避免“旧 nonce 被缓存后复用”

所以从架构角度看，这个方案的核心能力是：

**在反向代理层动态注入每请求唯一的安全令牌（nonce），而不改上游应用代码。**

---

## 2. 这个方案实际上实现了什么功能

### 2.1 动态响应内容改写

`ngx_http_sub_module` 的本质能力是：

- 对响应体做字符串替换
- 典型用于改写 HTML、JS、CSS、JSON 中的某些固定占位符

在你的场景里，它不是简单的文本替换，而是：

- 把静态模板占位符
- 变成每请求动态值

### 2.2 在代理层实现“每请求唯一 nonce”

如果你在 HTML 中有：

```html
<script nonce="__CSP_NONCE__">
```

然后在 Nginx 中做：

```nginx
sub_filter "__CSP_NONCE__" $request_id;
```

那么返回给浏览器时会变成：

```html
<script nonce="8d6b7f1e9a4f4c8b9e3d2f7a1c0b5e6f">
```

这就等价于：

- 每次请求得到不同的 nonce
- 浏览器只允许带这个 nonce 的 inline script 执行

### 2.3 代理层与应用层解耦

这个方案最大的现实价值之一是：

**不需要应用自己生成 nonce，也不需要应用自己改模板逻辑。**

也就是说，你把一部分原本应该由应用层完成的安全输出责任，转移到了 Nginx 层。

---

## 3. 架构上适合什么场景

### 场景 A：遗留系统无法快速改代码

如果你的应用：

- 历史包袱重
- 模板引擎不好改
- 由第三方团队维护
- 短期内不方便上线 CSP nonce 能力

那么在代理层做 `sub_filter` 是一个现实可落地的过渡方案。

### 场景 B：统一入口网关要为多个应用补安全能力

如果 Nginx 是统一入口层，后面挂多个遗留前端系统，那么这个方案可以作为：

- 平台层补丁
- 网关层统一能力

让多个系统先具备基本 nonce 注入能力。

### 场景 C：过渡期方案

这个方案非常适合作为：

- CSP 上线前的过渡方案
- 旧系统迁移到正确实现前的临时兜底

但它通常不应该被误认为是长期最优架构。

---

## 4. 它能带来的价值

### 4.1 快速接入 CSP nonce

最大的直接价值是：

- 你可以较快把 nonce 注入链路打通
- 而不必等所有应用改造完

### 4.2 降低应用改造成本

如果由应用自己做 nonce，通常要处理：

- 模板渲染
- 请求上下文传递
- CSP header 拼接
- 前端标签改造

而 Nginx 方案可以先把其中一部分收敛到网关层。

### 4.3 平台统一控制

如果是平台团队维护入口层，这样做有利于：

- 统一策略
- 统一发布
- 统一回滚

---

## 5. 这个方案的架构代价

这部分很重要，因为它决定了这个方案适不适合长期使用。

### 5.1 Nginx 从“转发器”变成“响应内容处理器”

一旦启用 `sub_filter`，Nginx 不再只是：

- TLS termination
- reverse proxy
- routing

它开始承担：

- 动态内容处理
- 安全令牌注入
- 输出内容一致性责任

这意味着你的入口层复杂度明显上升。

### 5.2 响应路径更重

`sub_filter` 要修改响应体，就意味着：

- 不能再完全透明透传
- 要参与 body 处理
- 对大响应体会有额外 CPU 和内存开销

所以这不是“零成本开一个模块”。

### 5.3 缓存策略会被改变

你给出的：

```nginx
proxy_cache_bypass 1;
proxy_no_cache 1;
add_header Cache-Control "no-cache, no-store, must-revalidate" always;
```

本质上是在告诉系统：

- 这类响应不能安全地共享缓存
- 因为每次响应里的 nonce 都不同

所以你实际上是用“放弃缓存效率”换“每请求唯一 nonce”。

---

## 6. 安全上它到底做对了什么

### 6.1 为什么 nonce 要每请求唯一

CSP nonce 的核心要求之一就是：

- 不可预测
- 每请求不同

这样浏览器只信任当前响应里明确授权的 inline script。

如果 nonce 被复用，安全收益会打折。

### 6.2 `$request_id` 作为 nonce 的意义

你提到：

- `$request_id` 是 Nginx 为每个请求生成的唯一值
- 通常是 32 位十六进制字符串

从工程实践角度，它具备这些优点：

- 每请求不同
- 平台层直接可用
- 不需要额外引入随机生成逻辑

所以它可以作为一个**工程上可接受的 nonce 值来源**。

### 6.3 为什么要禁用缓存

如果带 nonce 的页面被缓存，然后下一个用户拿到同样页面：

- 页面里的 nonce 仍然是旧值
- 可能和当前请求上下文不一致
- 更严重时会导致 CSP 行为异常

因此配合：

```nginx
proxy_cache_bypass 1;
proxy_no_cache 1;
Cache-Control: no-store
```

在逻辑上是成立的。

---

## 7. 风险与潜在隐患

### 7.1 最大风险：你只替换了 HTML nonce，但没有同步生成正确的 CSP header

这是最关键的风险。

如果你只是把页面里的：

```html
nonce="__CSP_NONCE__"
```

替换成 `$request_id`

但 `Content-Security-Policy` 头里没有同步包含：

```text
script-src 'nonce-同一个值'
```

那么浏览器并不会因为标签上有 nonce 就自动信任它。

也就是说：

**页面里的 nonce 和 CSP header 里的 nonce 必须完全一致。**

如果只是 body 替换，而 header 没跟上，这个方案是不完整的。

### 7.2 `$request_id` 的随机性不是“密码学设计”的 nonce 系统

这里要实话实说：

- `$request_id` 的工程唯一性很好
- 但它不是专门为 CSP nonce 设计的密码学随机源

在大多数企业内网/普通 Web 场景中，通常够用；
但如果你要追求严格安全边界，应用自己生成高质量随机 nonce 仍然更标准。

### 7.3 `sub_filter_once off` 可能带来过度替换

你配置了：

```nginx
sub_filter_once off;
```

这表示：

- 页面里所有匹配的 `__CSP_NONCE__` 都会被替换

好处是：

- 多个 `<script>` / `<style>` 标签都能一起替换

风险是：

- 如果响应内容中其他地方也出现了相同字符串
- 也会被替换

所以占位符一定要足够独特，不要用通用词。

### 7.4 可能影响压缩、缓存和流式输出

响应体改写通常意味着你要特别注意：

- `gzip` / `brotli`
- `Content-Length`
- chunked transfer
- upstream compression

如果上游已经压缩，Nginx 可能无法直接替换，或者需要额外配置解压/重写路径。

### 7.5 对大页面和高并发场景有性能影响

因为每次都要扫描和替换响应体，所以：

- HTML 越大，成本越高
- QPS 越高，Nginx worker CPU 压力越大

这不是一个“完全免费”的安全能力。

### 7.6 禁用缓存可能影响前端性能

如果你的页面原本可缓存，启用 nonce 注入后：

- 入口缓存命中下降
- 回源增多
- 首屏性能可能变差

这在高流量前端页面场景下尤其要评估。

---

## 8. 它适合做什么，不适合做什么

### 适合

- 作为遗留系统 CSP 改造的过渡方案
- 平台层快速验证 nonce 机制
- 对动态 HTML 页面做统一安全补丁

### 不适合

- 长期替代应用层正确实现 CSP nonce
- 用在大体积响应、高缓存依赖页面、超高并发入口而不做性能评估
- 只做 body 替换、不做 CSP header 同步

---

## 9. 你这段配置在架构上意味着什么

### 配置

```nginx
sub_filter_once off;
sub_filter "__CSP_NONCE__" $request_id;

proxy_cache_bypass 1;
proxy_no_cache 1;
```

### 架构含义

这实际上表示：

1. 入口层把占位符模板动态变成每请求 nonce
2. 为保证 nonce 正确性，主动放弃共享缓存
3. Nginx 不再只是代理，而是承担安全输出编排职责

### 换句话说

你是在用 Nginx 做一层“安全模板后处理引擎”。

这是可行的，但它已经超出了传统“反代配置”的范畴。

---

## 10. 推荐的落地判断

### V1：如果目标是快速验证

可以使用这个方案，但前提是：

- 占位符足够独特
- HTML 中的 nonce 替换和 CSP header 中的 nonce 一致
- 明确这类响应不能缓存
- 先在低风险页面验证

### V2：如果目标是长期生产化

更推荐的终态通常是：

- 应用层生成 nonce
- 模板引擎直接渲染 nonce
- Nginx 只负责透传和附加统一 header

这样职责边界更清楚，也更容易维护。

---

## 11. 最关键的结论

### 它实现了什么

它实现的是：

**由 Nginx 在响应返回阶段，为页面动态注入每请求唯一 nonce，从而帮助 CSP 生效。**

### 它的价值

它最大的价值是：

**不用改上游应用，就能较快补上一层 CSP nonce 能力。**

### 它的代价

它的代价是：

- 增加入口层复杂度
- 增加响应体处理开销
- 牺牲缓存能力
- 要求 body 和 CSP header 保持严格一致

### 它的风险

最大的风险不是模块本身，而是：

**只替换了响应体中的 nonce，占位了 HTML，但没有同步维护正确的 CSP header。**

如果这一步没做完整，这个方案看起来“像是有 nonce”，实际上并没有真正完成 CSP 安全闭环。

---

## 12. 建议

如果你现在要推进这个能力，我建议你优先确认这四件事：

- [ ] 当前 Nginx 是否已编译 `ngx_http_sub_module`
- [ ] CSP header 是否能和 `$request_id` 保持同值同步输出
- [ ] 目标页面是否允许关闭缓存
- [ ] 高并发和大页面场景下是否评估过性能影响

如果这四件事都能闭环，这个方案可以作为一个现实可落地的过渡设计。
